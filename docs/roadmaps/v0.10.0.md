## Why now / why this shape

With correlation made real (v0.8.0) and smart (v0.9.0), this is the first **breadth** version of the
post-analysis arc. It completes the operational-signal coverage the analysis found partial:

- **#1 is only ~25% covered:** only the *cron-parado* sub-variant, only the *disconnected* shape.
  The *alive-but-stalled* shape and the poison-message / stuck-lock / kill-switch sub-variants are
  invisible, partly because the **progress axis** (ack rate / counter deltas) is never shipped, and
  partly because there is **no standalone** queue/consumer diagnosis — everything hangs off
  `CronHealthPattern`, so a dead consumer with a healthy cron produces nothing.
- Two catalog signals flagged as **easy wins** (`maintenance_flag`, `schema_drift`) are still not
  shipped.
- **#4/#6 search** is covered only on the DB/mview side; the OpenSearch/Elasticsearch engine half is
  missing.

## Version goal

**Complete the operational-signal coverage:** ship the queue/consumer progress axis and a standalone
diagnosis so #1's remaining shapes/sub-variants become detectable; add the two easy Wire-Up signals;
and cover the search-engine cluster half of #4/#6.

## v0.10.0 success definition

Against the live lab:

1. A dead/stalled consumer is diagnosed **even when cron is healthy** (standalone), and the
   *alive-but-stalled* shape (consumers > 0, ack rate = 0) fires from the new progress axis.
2. `maintenance_flag` and `schema_drift` ship and are read as canonical signals.
3. Search staleness is diagnosable from **both** the `catalogsearch_fulltext` indexer/mview side and
   the OpenSearch/Elasticsearch cluster (`_count` vs catalog size, cluster status).

## Implementation rules

- **Ship raw state; diagnose in Provado.** New signals emit raw readings; dwell/baseline/weight stay
  on the Provado side, reusing the v0.6.0/v0.9.0 seams.
- **Wire-Up before Instrument.** `maintenance_flag` and `schema_drift` are plain reads for
  `provado-ship.php`; the search-engine cluster half may straddle Instrument.
- **Honest lab caveats.** Note plainly which shapes are exercised organically vs. by a
  forced/synthetic condition (as v0.6.0/v0.7.0 did).
- Every source-code phase is green on the lab server + a `/code-review` pass (per `CLAUDE.md`).

## Phase 1 — Progress axis + standalone diagnosis

**Status:** [ ]

> Core signal + pattern change — **plan approval required** per `CLAUDE.md`.

**Goal:** Make failure mode #1 detectable beyond the cron-attributed disconnected shape. Ship the
**progress axis** (ack/deliver-counter deltas between polls; DB-queue `updated_at` advancement) so
*alive-but-stalled* is visible, and add a **standalone** queue/consumer diagnosis that fires without
requiring a degraded cron.

**Deliverables (items):**

1. [ ] **Ship the progress axis** on `queue_backlog`/`consumer_liveness` (ack/deliver rate or
   counter deltas; DB-queue `in_progress` aging past `retry_inprogress_after`). Reconcile the
   `signal-shipping.md` catalog's `ack_rate` claim with what now actually ships.
2. [ ] **Standalone queue/consumer pattern** that diagnoses a dead/stalled consumer with a healthy
   cron (poison message stuck at `in_progress`/`error`; stuck lock — `running`=held but no progress;
   the `only_spawn_when_message_available` "empty queue is normal" nuance). It must dedupe against
   the `CronHealthPattern` collapse so a cron-caused death is still attributed once, not twice.

## Phase 2 — Easy Wire-Up wins

**Status:** [ ]

> Shipper + reader change; the diagnoses are small.

**Deliverables (items):**

1. [ ] **`maintenance_flag`** — one `stat` on `var/.maintenance.flag`, shipped as `present` (0/1).
   Diagnosis: flag present while traffic continues is a likely leftover-after-deploy (distinct from a
   real outage). Cheap signal, high consequence.
2. [ ] **`schema_drift`** — `setup_module` schema vs data version per module, shipped as
   `mismatched_modules`. Handle the known MySQL 8.0.x false-positive noise (diff versions rather than
   trusting the binary `setup:db:status`).

## Phase 3 — Search-engine cluster half

**Status:** [ ]

> New collector that may straddle Instrument — **plan approval required** per `CLAUDE.md`.

**Goal:** Cover the engine side of #4/#6 that the DB/mview `indexer_status` cannot see: an
OpenSearch/Elasticsearch cluster that is unreachable/yellow/red or whose doc count diverges from the
catalog while the DB indexer reports valid.

**Deliverables (items):**

1. [ ] **Ship search-engine health**: cluster status + `_count` vs catalog size, joined to the
   `catalogsearch_fulltext` indexer/`_cl` backlog, so a DB-side stall and an engine-side failure are
   distinguishable.
2. [ ] **Diagnose stale/empty search** from the joined signal (engine unreachable/empty while DB
   indexer valid = search returns nothing though products exist), dwell-qualified.
