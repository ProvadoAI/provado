## Why now / why this shape

After breadth on the operational-health signals (v0.10.0), this version reaches the **revenue-
integrity** failure modes and completes the **change spine**:

- **#6 (silent order loss)** and **#7 (order created, payment never captured)** are the failure
  modes that lose money directly, and both already have drafted simulations
  (`SIM-PAY-001`/`SIM-PAY-002`). Their Magento-side signals (orphaned quote fingerprint, uncaptured
  auth aging) are readable from operational state; full reconciliation against the PSP is a further,
  gated step.
- **#5 (deploy/config regression)** is half-built: `config_change` covers the marker-less
  `core_config_data` surface, but the **New Relic deploy markers** (`FROM Deployment`) — the spine
  that timestamps "what changed" — are not read yet.

## Version goal

**Ship the order/payment-integrity signals (Magento side) and the New Relic deploy-marker reader**,
so the money failure modes are diagnosable and every onset can be stamped against a real deploy
marker, not only against config churn.

## v0.11.0 success definition

1. A finished checkout with **no matching `sales_order`** (orphaned quote: `reserved_order_id` set,
   `is_active = 1`) is diagnosed as **#6**, dwell-qualified.
2. An order with an **authorization but no capture child transaction**, aging past its window, is
   diagnosed as **#7**.
3. Provado reads **New Relic deploy markers** and stamps symptom/cause onsets against the nearest
   marker (completing the change spine), where the merchant environment provides them.

## Implementation rules

- **Provado stays remote / ship raw state.** The Magento-side integrity signals ship raw
  fingerprints and aging; diagnosis stays in Provado.
- **Honest gating.** The lab is Magento Open Source on-prem with **no code deploys**, so the deploy-
  marker reader is verified against recorded/synthetic markers and belongs to Adobe Commerce Cloud /
  change-tracking merchants — stated plainly (mirrors v0.6.0/v0.7.0 honesty notes).
- Every source-code phase is green on the lab server + a `/code-review` pass (per `CLAUDE.md`).

## Phase 1 — Order/payment integrity (Magento side)

**Status:** [ ]

> New signal + patterns — **plan approval required** per `CLAUDE.md`. Injection designs already
> drafted in `SIM-PAY-001` / `SIM-PAY-002`.

**Goal:** Diagnose the two revenue-loss modes from Magento operational state, without yet requiring
a PSP source.

**Deliverables (items):**

1. [ ] **`order_integrity` — silent order loss (#6):** ship the quote fingerprint (`quote` rows with
   `reserved_order_id` set and `is_active = 1` but **no matching `sales_order`**), as
   `orphaned_quotes` with age. Diagnose the contradiction, dwell-qualified, stamped against the
   change/config timeline. Cross-reference the 1213/1205 deadlock signal where present.
2. [ ] **`order_integrity` — uncaptured payment (#7):** ship `sales_payment_transaction` rows with an
   authorization but no capture child, aging past the auth window (`uncaptured_aging`). Diagnose as
   authorized-not-captured. Note: full reconciliation against the PSP (Stripe/Adyen) is **deferred**
   (needs a PSP source, out of lab).

## Phase 2 — New Relic deploy markers (change spine)

**Status:** [ ]

> New reader mode on the existing NerdGraph client — **plan approval required** per `CLAUDE.md`.
> **Gated:** the lab has no deploys, so verify against recorded/synthetic markers.

**Goal:** Read New Relic change-tracking **Deployment** events (`FROM Deployment` / change tracking)
so Provado can stamp an onset against "what changed", completing the change spine that `config_change`
only half-covers.

**Deliverables (items):**

1. [ ] A **deploy-marker reader mode** on `NerdGraphClient` (`FROM Deployment`), mapping markers to
   canonical change signals within the window — no vendor shape crossing the seam. Verified via a
   recorded/synthetic marker (contract test), not live capture.
2. [ ] **Stamp onsets against the nearest marker** in the diagnosis evidence (alongside the existing
   `config_change` stamp), so a verdict carries a likely trigger — deploy *or* config edit.
