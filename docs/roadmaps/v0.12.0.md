## Why now / why this shape

After v0.9 (smart correlation), v0.10 (operational-signal completeness) and v0.11 (money modes,
Magento side), the remaining catalog reds are **commerce-logic** modes. All three are the same
build shape as everything already shipped — read-only SELECTs shipped through the existing
`ProvadoSignal` contract, diagnosed by a new pattern reusing the dwell/baseline (v0.6.0) and
confidence-weight (v0.9.0) seams:

- **#2 (inventory drift):** the reservation ledger is append-only and fails silently — a lost
  compensation is a row never written, so nothing throws. Adobe's own remediation is a manual CLI
  run. The detection is a scheduled version of `inventory:reservation:list-inconsistencies` logic:
  a final-state order whose `inventory_reservation` sums ≠ 0.
- **#3 (promotion/price-rule):** the Admin surfaces *status* (`is_active`), never *effect*. A rule
  can be active, in window, in scope, un-capped — and still apply nothing (stalled materialization,
  usage caps, priority knock-out). The contradiction is status-vs-effect.
- **#10 (per-region config break):** a one-market break hides in aggregate metrics. The detection
  is segmentation: per-`store_id` checkout success against its own baseline, stamped against the
  scoped `core_config_data` change.

## Version goal

**Diagnose the three commerce-logic failure modes (#2, #3, #10) on the live lab**, each as a new
signal + pattern on the existing seams, each reproduced via its simulation before the coverage map
row moves.

## v0.12.0 success definition

Against the live lab:

1. An order in a **final state** whose `inventory_reservation` sequence sums ≠ 0 is diagnosed as
   inventory drift (#2), with the exact `order:sku:qty:stock` line as evidence, dwell-qualified.
2. A rule that is active, in window, in scope and un-capped — but has **no current materialized
   row** (catalog rule) or whose **measured effect collapsed** against its learned baseline (cart
   rule) — is diagnosed as #3. Honest cold-start caveat: a brand-new rule has no baseline; the
   strongest signal is an established rule that stops.
3. A checkout/tax break confined to **one market** is diagnosed as #10 by per-`store_id`
   segmentation (not aggregate), stamped against the scoped `core_config_data.updated_at` change.

## Implementation rules

- **Ship raw state; diagnose in Provado.** New signals emit raw readings; dwell/baseline/weight
  stay on the Provado side, reusing the v0.6.0/v0.9.0 seams.
- **All reads.** Provado never writes compensations (#2), never re-applies rules (#3), never edits
  config (#10) — it names the exact line/row for the engineer.
- **Simulation before coverage.** Each mode is reproduced via its simulation (`SIM-ERP-002`;
  `SIM-PROMO-001` and `SIM-CONFIG-002` authored per `docs/simulations/PLAN.md` Wave 3 inside the
  respective phase) before its `docs/COVERAGE.md` row moves.
- Every source-code phase is green on the lab server + a `/code-review` pass (per `CLAUDE.md`).

## Phase 1 — Inventory drift (#2)

**Status:** [ ]

> New signal + pattern — **plan approval required** per `CLAUDE.md`. Injection design in
> `SIM-ERP-002`.

**Goal:** Detect the silent, append-only reservation drift: a final-state order still holding a
non-zero reservation sum (false out-of-stock or oversell), before a customer complains.

**Deliverables (items):**

1. [ ] **Ship `inventory_integrity`:** per-(sku, stock_id, object_id) non-zero
   `inventory_reservation` sums for orders in a final state (the `list-inconsistencies` logic on a
   schedule), plus presence/liveness of the `inventory.reservations.updateSalabilityStatus`
   consumer (leading indicator; reuse the `consumer_liveness` mechanism).
2. [ ] **Diagnose drift:** a pattern that fires on the contradiction ("order N is Complete but
   holds −X unreleased reservation on stock S"), dwell-qualified, stamped against the change/config
   timeline, cross-referencing the salability consumer's state. Evidence carries the exact
   `order:sku:qty:stock` line the engineer needs.

## Phase 2 — Promotion / price-rule effect collapse (#3)

**Status:** [ ]

> New signal + pattern — **plan approval required** per `CLAUDE.md`. `SIM-PROMO-001` is authored
> in this phase (PLAN.md Wave 3).

**Goal:** Detect a rule that reads healthy (`is_active=1`, in window, in scope, not capped) while
its effect has disappeared — the status-vs-effect gap the Admin has no surface for.

**Deliverables (items):**

1. [ ] **Ship `promo_health`:** rule status/dates/scope/caps (`salesrule`/`catalogrule` + scope and
   usage tables), catalog-rule **materialization** (current `rule_date` row present in
   `catalogrule_product_price`), and the **effect series** (`salesrule_coupon_aggregated`
   `discount_amount`/`coupon_uses`).
2. [ ] **Diagnose the contradiction:** active + in window + in scope + not capped, yet no current
   materialized row (catalog) or effect collapsed to zero against the learned baseline (cart) —
   qualified by the materialization pipeline's health (indexer/cron, already shipped) and stamped
   against the change timeline. Cold-start (no baseline) demoted honestly.

## Phase 3 — Per-region config break (#10)

**Status:** [ ]

> Extends an existing signal + new pattern — **plan approval required** per `CLAUDE.md`.
> `SIM-CONFIG-002` is authored in this phase (PLAN.md Wave 3).

**Goal:** Surface a one-market break immediately by segmenting per store instead of averaging it
away, and attribute it to the scoped config change that caused it.

**Deliverables (items):**

1. [ ] **Ship the scoped surface:** extend `config_change` with scope/path detail (which
   `core_config_data` row changed, for which `scope`/`scope_id`, mapped to website/store code), and
   ship **per-store checkout activity** (the live `order_activity` already carries per-status
   counts per store — segment it by `store_id` as its own baseline series).
2. [ ] **Diagnose the localized break:** market X's order/checkout profile breaks its own baseline
   while other stores hold, stamped against the matching scoped config change
   (`payment/<method>/active` for `scope_id=N`, tax-relevant paths), emitting one verdict naming
   the market and the exact row.
