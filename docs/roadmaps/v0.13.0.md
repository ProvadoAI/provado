## Why now / why this shape

v0.12 closes the commerce-logic reds; what remains are the modes that need **a new source or lab
infra** rather than another SELECT:

- **#8 (cross-system sync)** has a pattern and a reproduced simulation, but is fixture-only: the
  live REST adapter emits only `order_activity`, never the types the pattern requires. It needs the
  real integration surfaces — `magento_operation`/`magento_bulk`, the `async.operations.all`
  consumer, and the lab's Odoo as the ERP side.
- **#12 (cacheability)** needs the FPC/Varnish hit-ratio, which no shipper reads today
  (`cache_validity` is invalidation, not cacheability).
- **#6/#7** are diagnosed Magento-side in v0.11 but stop at "the ledger says so"; the PSP is the
  ground truth. The lab's **Stripe test mode** (per `SIM-PAY-001`'s environment) makes the
  reconciliation buildable now — it was deferred in v0.11 only as "out of lab", which Stripe test
  mode resolves.
- **#11 (measurement/tag)** gets its **decided verification path (2026-07-01): a real GA4 stack on
  the lab** (GA4 test property + gtag/GTM on the lab storefront), so the narrow Provado slice —
  GA4-vs-order-ledger divergence stamped against the change timeline — is verified live, not
  synthetically. Provado does **not** build general tag governance (ObservePoint-class scope stays
  out, per the catalog's honest framing).

## Version goal

**Complete the full-catalog coverage goal:** wire #8 to live integration state, ship cacheability
(#12), reconcile #6/#7 against Stripe test mode, and diagnose #11's narrow slice on a real GA4
stack — leaving only #9 gated by explicit decision.

## v0.13.0 success definition

Against the live lab:

1. An inbound async/bulk push that is *accepted but never completes* (operations stuck at
   Open/failed, or the consumer dead) is diagnosed as #8 from live `magento_operation`/
   `magento_bulk` + consumer state — no fixtures — reproduced via `SIM-ERP-003`.
2. A page type whose FPC/cache hit ratio steps down while responses stay 200 is diagnosed as #12,
   stamped against the change timeline (the `cacheable="false"` / no-cache-header blast radius).
3. A Stripe-test capture with **no matching `sales_order`** confirms #6, and an authorization the
   gateway shows expired/uncaptured confirms #7 — the reconciliation join, not just the
   Magento-side fingerprint.
4. A GA4-purchases-vs-`sales_order` divergence on the lab storefront is diagnosed as #11 (narrow
   slice), stamped against the deploy/config timeline.

## Implementation rules

- **Ship raw state; diagnose in Provado.** Same seams as always (dwell/baseline v0.6.0, weight
  v0.9.0, change-stamp).
- **New sources land behind the existing seams** (`SourceAdapter` + canonical `Signal`); no vendor
  shape crosses the adapter boundary (Stripe, GA4, Odoo included).
- **Key on terminal state, not receipts** (#8): HTTP 202 / `accepted` is explicitly not success.
- **Honest scope on #11:** the diagnosis is the order-ledger-anchored divergence + change-stamp
  slice only — no general tag/consent governance claims.
- **Infra items are deliverables too:** the GA4 stack (Phase 4 item 1) is part of the phase, with
  its setup documented so it is reproducible.
- Every source-code phase is green on the lab server + a `/code-review` pass (per `CLAUDE.md`).

## Phase 1 — #8 live wiring (ERP / integration surfaces)

**Status:** [ ]

> New signal + source wiring — **plan approval required** per `CLAUDE.md`. Reproduction via
> `SIM-ERP-003`.

**Goal:** Make `CatalogFeedSyncFailurePattern`'s mode diagnosable from live state instead of
fixtures: the acceptance-is-not-success boundary (`magento_operation` status 2/3/4 piling up,
`async.operations.all` dead) and the ERP side of the join.

**Deliverables (items):**

1. [ ] **Ship the integration surfaces:** `magento_operation`/`magento_bulk` status counts (per the
   verified enum: 1 Complete / 2 Failed-retriable / 3 Failed-not-retriable / 4 Open / 5 Rejected)

2. [ ] **Wire the ERP side and rewire the pattern:** Odoo (XML-RPC, read-only) as a live source

   and change-stamped; reproduce via `SIM-ERP-003`.

## Phase 2 — Cacheability broken by deploy (#12)

**Status:** [ ]

> New signal + pattern — **plan approval required** per `CLAUDE.md`. `SIM-CACHE-001` is authored
> in this phase (PLAN.md Wave 2).

**Goal:** Detect the "uncacheable is not an error" failure: hit ratio steps down, every response
is a healthy 200, and the cause is a deploy-introduced `cacheable="false"` or no-cache header.

**Deliverables (items):**

1. [ ] **Ship the hit-ratio signal** from the lab's cache front (Varnish `varnishstat` or built-in
   FPC), per page-type where readable, plus the cacheability probe (`X-Magento-Cache-Debug` /
   `X-Cache` HIT/MISS on representative routes). Honest caveat: on ACC the same signal shape comes
   from the Fastly Stats API — that adapter half stays gated with #9's infra.
2. [ ] **Diagnose the step-down:** ratio breaks its learned baseline while responses stay 200,
   stamped against the change timeline; where the layout/config surface is readable, name the
   offending handle/header (the #9041 blast-radius case: a `default`-handle block that never even
   renders).

## Phase 3 — PSP reconciliation (Stripe test mode) — complete #6/#7

**Goal:** Add the gateway ground truth the v0.11 fingerprints lack: a captured PSP transaction

**Deliverables (items):**
