## Version goal

**From fixtures to a live, installable Tier 0 connector.**

v0.1.0 proved the ingestion → normalization → correlation → diagnosis → incident loop
end-to-end, but only over fixtures and barely wired into Laravel. v0.2.0 makes that loop
**real and usable inside a Laravel application**: wire the package into the container,
give it a runnable entry point, replace the fixture clients with authenticated New Relic
(NerdGraph) and Adobe Commerce (REST) clients behind the existing adapter seam, persist
signals, and add one more narrow Tier 0 pattern.

This follows `ARCHITECTURE_DIRECTION_SOURCE.md`: the connector layer is "the foundation
everything else sits on." The source adapter, retry, and observability seams built in
v0.1.0 are waiting to receive real clients.

## v0.2.0 success definition

The version is successful when the system can:

1. Be installed into a Laravel app and resolve a fully-wired `DiagnosticPipeline` from the container.
2. Run the full diagnostic loop from a single artisan command and render a readable incident report.
3. Fetch real Tier 0 data from New Relic (NerdGraph) and Adobe Commerce (REST) using configured
   credentials, normalized through the existing canonical model.
4. Detect Adobe Commerce Cloud tier (Pro vs Starter) and adjust the expected signal envelope.
5. Persist canonical signals to a database-backed store via the existing storage seam.
6. Run at least two narrow Tier 0 diagnostic patterns.
7. Fall back cleanly to fixtures when no credentials are configured, with no real calls.

## Implementation rules

- Keep New Relic swappable: no vendor response shapes leak above the adapter seam — everything
  downstream still reasons over the canonical signal model.
- Reuse the v0.1.0 seams (`SourceAdapter`, `RetryPolicy`, `PipelineObserver`, `SignalStore`,
  `SignalStoreFactory`); do not fork them.
- Real provider calls only behind configured credentials; the default/test path stays fixtures.
- Stay narrow: do not build pattern-library depth, revenue attribution, economic-impact
  estimation, prediction, or external-market signals — those remain deferred north-star items
  (see `opportunities.md`).
- Every source-code phase is a reviewed PR with a `/code-review` pass (per `CLAUDE.md`).

## Phase 1 — Laravel container wiring

**Status:** [ ]

**Goal:** Make the package usable from a Laravel app by building `ProvadoConfig` from
`config/provado.php` and binding the full pipeline graph into the container, without
adding new diagnostic capability.

**Deliverables:**

- A factory/binding that builds an immutable `ProvadoConfig` from `config('provado')`
- `ProvadoServiceProvider` binds `SourceAdapterRegistry`, `SignalStoreFactory`,
  `DiagnosticPatternRegistry`, `IncidentReportBuilder`, and a resolvable `DiagnosticPipeline`
- A `PipelineObserver` binding that defaults to `NullPipelineObserver`, optionally a
  `PsrLoggerObserver` over Laravel's logger when enabled in config
- Fixture adapters remain the default registered sources
- `orchestra/testbench` added as a dev dependency; tests assert the container resolves a
  working pipeline and that config maps correctly

---

## Phase 2 — Runnable diagnose command (Alpha demo)

**Status:** [ ]

**Goal:** Provide a single runnable entry point that executes the whole loop and renders a
readable incident report — the demonstrable Alpha flow.

**Deliverables:**

- `provado:diagnose` artisan command accepting a time window (with sensible defaults)
- Runs the container-resolved `DiagnosticPipeline` and renders output via `IncidentReportRenderer`
- A console-backed `PipelineObserver` that prints stage progress and the run diagnostics summary
- Works on fixtures with no credentials by default
- Tests covering command output for the populated-report and no-report cases

---

## Phase 3 — HTTP source-client foundation

**Status:** [ ]

**Goal:** Establish the shared HTTP plumbing real source clients need, behind clean interfaces,
without yet calling any real provider by default.

**Deliverables:**

- A small HTTP client seam (timeouts, auth headers, JSON handling) used by real adapters
- Transport/HTTP failures mapped to `SourceFetchError` with correct `retryable` classification
  (timeouts/5xx/429 retryable; 4xx auth/config not), so the existing `RetryPolicy` drives retries
- Rate-limit awareness (honor `Retry-After`/429 as retryable)
- No behavior change to fixture adapters; unit tests use a fake/recorded transport, no live calls

---

## Phase 4 — New Relic NerdGraph client

**Status:** [ ]

**Goal:** Fetch real New Relic observability signals via NerdGraph behind the existing
`SourceAdapter` seam, normalized through the canonical model.

**Deliverables:**

- A NerdGraph client (GraphQL) authenticated with `account_id` + `api_key` from config
- New Relic adapter path that maps NerdGraph responses via `NewRelicPayloadMapper`
  (extended as needed) into canonical signals — no vendor shapes leak downstream
- Retry via `RetryPolicy`; clean fallback to the fixture client when credentials are absent
- Contract tests against recorded NerdGraph responses (no live credentials in CI)

---

## Phase 5 — Adobe Commerce REST client + Cloud tier detection

**Status:** [ ]

**Goal:** Fetch real Adobe Commerce commerce-state signals via REST, accounting for the
Cloud tier whose signal envelope differs.

**Deliverables:**

- An Adobe Commerce REST client authenticated with `base_url` + `access_token`
- **Cloud tier detection (Pro vs Starter)** that adjusts which signals are expected/collected
- Adobe adapter path mapping REST responses via `AdobeCommercePayloadMapper` into canonical signals
- Retry via `RetryPolicy`; clean fallback to the fixture client when credentials are absent
- Contract tests against recorded REST responses, including both tiers

---

## Phase 6 — Signal persistence

**Status:** [ ]

**Goal:** Persist canonical signals to a database via the existing `SignalStore` seam so runs
survive beyond a single process, without changing pipeline logic.

**Deliverables:**

- A database-backed `SignalStore` implementing the existing interface, plus a matching
  `SignalStoreFactory`
- Migrations under `database/migrations` for the signal store
- Config switch selecting in-memory (default) vs database store
- Tests for save/query/lookup parity with `InMemorySignalStore`

---

## Phase 7 — Second Tier 0 diagnostic pattern

**Status:** [ ]

**Goal:** Broaden diagnostic coverage with one more deterministic pattern fully diagnosable
from Tier 0 signals, keeping the set deliberately narrow.

**Deliverables:**

- One new `DiagnosticPattern` under `src/Patterns/` (candidate: a payment-config / 3DS
  regression or order-sync/indexer-backlog pattern, per the architecture doc's Tier 0 examples)
- Registered alongside the checkout-degradation pattern
- PHPUnit coverage for identity, support matching, evidence, recommendations, and severity

---

## Phase 8 — Hardening and architecture docs

**Status:** [ ]

**Goal:** Polish real-run behavior and document the now-working Alpha architecture.

**Deliverables:**

- Backoff/rate-limit polish surfaced through run diagnostics and the diagnose command
- `docs/ARCHITECTURE.md` filled in to describe the working Alpha architecture (layers, seams,
  data flow) instead of pointing only at the direction doc
- README/package docs updated for installation, configuration, and the diagnose command

## Deferred on purpose (still out of scope for v0.2.0)

These remain behind clean interfaces until a real merchant or design partner pulls them forward
(see `docs/ARCHITECTURE_DIRECTION_SOURCE.md` Part 2 and `opportunities.md`):

- Full pattern library and dimension-specific diagnostic logic
- Revenue attribution, economic-impact estimation, and performance-fee logic
- Revenue-risk prediction / trajectory analysis and external-market signals
- Tier 1+ sources (CrUX/RUM, payment processors, GA4/Google Ads, AI-discovery, edge/CDN logs)
- Real payment processor integrations
- Multi-tenant SaaS UI and automated remediation
