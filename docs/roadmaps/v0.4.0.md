## Version goal

**Promote the deferred Tier 0 integration: read real signals from New Relic (NerdGraph/NRQL)
and Magento (Adobe Commerce REST), normalized into the canonical model, without changing
anything downstream of the adapter seam.**

v0.3.0 laid every seam a real client needs — the provider-agnostic `HttpClient`
(`LaravelHttpClient` + `FakeHttpClient`), transport-failure mapping to `SourceFetchError`,
credential-driven client selection with clean fixture fallback, and the recorded-response
contract-test harness. v0.4.0 fills in the only thing still missing: the concrete credentialed
clients, the real-response mapping paths, edition detection, and a way to validate live
connectivity. The `NewRelicClient` and `AdobeCommerceClient` interfaces already exist as empty
seams; the adapters already select a credentialed client when credentials are present and fall
back to fixtures otherwise. This version makes those seams real against a live environment.

## Environment this version targets

The lab the live work is built and validated against:

- **New Relic** — APM data via NerdGraph (GraphQL + NRQL), authenticated with an account ID
  and a User API key.
- **Magento 2.4.9 Open Source** — self-hosted (CentOS Stream 9, Apache + PHP 8.3, MariaDB,
  OpenSearch, Redis, RabbitMQ). REST API authenticated with an integration access token.
  **Note:** this is Magento **Open Source on-prem**, *not* Adobe Commerce Cloud — so the
  deferred "Cloud tier detection (Pro vs Starter)" deliverable is reinterpreted here as
  **edition detection** (Open Source on-prem vs Adobe Commerce Cloud), with the lab landing in
  the on-prem envelope and the Cloud path designed but not exercised.

## v0.4.0 success definition

The version is successful when, against the live lab, the system can:

1. Fetch real New Relic signals via NerdGraph and normalize them into the canonical model.
2. Fetch real Magento commerce-state signals via REST, detecting that the instance is Open
   Source on-prem and collecting the matching signal envelope.
3. Validate connectivity and credentials for both sources via a dedicated command, without
   running the full pipeline.
4. Run `provado:diagnose` end-to-end over real data and render an incident report.
5. Exercise contract tests against **captured, redacted** real responses — no live credentials
   in CI.

## Implementation rules

- **Keep providers swappable:** no NerdGraph or REST response shape crosses the adapter seam —
  everything downstream still reasons over the canonical signal model. If a downstream layer
  needs to change to accommodate a real client, that is a smell that a vendor shape leaked up.
- **Reuse the existing seams** (`HttpClient`, `RetryPolicy`, `SourceFetchError`,
  `HttpSourceErrorFactory`, `SourceAdapter`, `PipelineObserver`, `SignalStore`,
  `RecordedResponseLoader`); do not fork them.
- **Secrets:** credentials are supplied only via environment (already wired in
  `config/provado.php`); never commit tokens or account IDs. Responses captured for contract
  tests must be **redacted** (reuse `ContextRedactor`) before being committed.
- **Retry + fallback:** real clients drive retries through the existing `RetryPolicy` and fall
  back cleanly to the fixture client when credentials are absent.
- **Stay narrow:** do not build pattern-library depth, revenue attribution, economic-impact
  estimation, prediction, or Tier 1+ sources — those remain deferred north-star items. Odoo
  (XML-RPC `provado_erp_faults`) is a new non-Tier-0 source and is **deferred to v0.5.0** as a
  new adapter to the canonical model.
- Every source-code phase is a reviewed PR with a `/code-review` pass (per `CLAUDE.md`).

## Phase 1 — New Relic NerdGraph client (live)

**Status:** [x]

**Goal:** Fetch real New Relic observability signals via NerdGraph behind the existing
`NewRelicClient` seam, over the existing `HttpClient`.

> Core source-adapter change — **plan approval required** per `CLAUDE.md` before implementing.

**Deliverables:**

- A `NerdGraphClient` implementing `NewRelicClient`: GraphQL POST to the NerdGraph endpoint,
  NRQL queries scoped by `account_id`, the User API key sent in the `API-Key` header, built on
  the injected `HttpClient` (no direct outbound I/O of its own).
- Transport/HTTP failures mapped via `HttpSourceErrorFactory` to `SourceFetchError` with correct
  `retryable` classification, so the existing `RetryPolicy` drives retries; rate limits honored.
- Wired into `NewRelicAdapter` so it is selected when `api_key` is present and falls back to the
  fixture client when it is absent (no behavior change to the fixture path).
- Container/config wiring so the client resolves from `config('provado')`; no live call until the
  adapter invokes it.
- Unit tests using `FakeHttpClient` (no live calls) covering request shape, auth header, error
  mapping, and fixture fallback.

## Phase 2 — Real-response New Relic mapping

**Status:** [x]

**Goal:** Extend `NewRelicPayloadMapper` to translate real NerdGraph/NRQL result shapes into
canonical signals, distinct from the fixture path.

> Core mapping change — **plan approval required** per `CLAUDE.md` before implementing.

**Deliverables:**

- A real-response mapping path that maps NRQL result rows into canonical `Signal`s, separate from
  the fixture-shaped path (generalize `rawPayloadReference` so it no longer hardcodes the
  `tests/Fixtures/...` location).
- No NerdGraph response shape leaks downstream — only canonical signals cross the adapter seam.
- PHPUnit coverage over representative recorded NRQL shapes (timestamps, severity, entity
  references, numeric metric attributes), including malformed-response handling.

## Phase 4 — Real-response Magento order mapping (per-status)

**Status:** [x]

**Goal:** Enrich the live REST order mapping so the commerce-state signal carries the full
order-state snapshot, correlatable from a single signal.

> Core mapping change — plan approved (single-signal model confirmed against the signal-catalog
> spec: "see all signals correlatively from one output").

**Deliverables:**

- [x] `AdobeCommercePayloadMapper::mapOrders` emits **one** `order_activity` signal per store
  carrying `order_count`, `gross_total`, and a per-status breakdown as `count_<status>` attributes
  (e.g. `count_processing`, `count_closed`). A single signal keeps entity+time correlation sharp
  (status counts are facets of one observation); per-status severity is left to the pattern layer.
- [x] PHPUnit coverage for the per-status breakdown and the missing/unknown-status path.
- [x] Live-validated against the lab: `order_count=3 gross_total=105.03 count_processing=2
  count_closed=1`.

**Deferred — edition detection.** The originally-scoped Open Source vs Cloud (Pro/Starter)
detection is **moved to the operational-state collector (v0.5.0)**. Rationale: the REST signals we
read are edition-agnostic (they don't change between editions), so detection adds nothing to
hardening them; edition-aware branching genuinely matters only for the deeper collector that
reads Magento operational state (whose access path differs by edition). Building it here would be
unused code. See `docs/ARCHITECTURE_DIRECTION_SOURCE.md` (rev v3).

## Phase 5 — Live source-inspection command (`provado:signals`)

**Status:** [x]

**Goal:** Make first contact with the live environment safe and debuggable — fetch each source and
show what it actually returns, without running the full correlation/pattern pipeline.

**Deliverables:**

- [x] A `provado:signals` artisan command that, for each enabled source, fetches over a time window
  (`--from`/`--to`/`--window`, or `--demo` over fixtures) and prints the canonical signals (metric
  values) per source in a table, plus a total. This both validates live connectivity and shows the
  data — broader than the originally-planned terse `provado:source:check` OK/fail check.
- [x] Per-source `SourceFetchError`s are surfaced inline (code, retryable, message), reusing the
  `HttpSourceErrorFactory`/`SourceFetchError` classification; credentials are never echoed.
- [x] Runnable via `vendor/bin/testbench provado:signals` (a `testbench.yaml` registers the
  provider; the lab `.env` supplies live credentials). The PHPUnit suite is isolated from any host
  `.env` via forced `<env>` defaults so tests never fire against a live environment.
- [x] Testbench-based tests covering the populated (`--demo`), no-enabled-sources, out-of-window,
  and invalid-input paths.
