## Version goal

**Ingest Magento operational signals remotely via the `ProvadoSignal` contract, and begin the
diagnosis layer (dwell + cross-signal correlation) on top.**

The only Provado-side ingestion is a generic `ProvadoSignal` reader over the existing NerdGraph
client; the merchant-side shippers are a versioned toolkit. The value Provado adds — and that
generic APM does not — is the diagnosis layer: learned-normal baselines, dwell-over-state, collapse
by Magento's dependency graph, and stamping onsets against the deploy/config timeline.

## v0.5.0 success definition

The system can, against the live lab:

1. Read `ProvadoSignal` custom events from New Relic via NerdGraph and map them to canonical signals
   — agnostic to which shipper produced them.
2. Ship at least `cron_health` and `indexer_status` from the lab through one shipper and see them
   surface in `provado:signals`.
3. Carry a versioned shipper toolkit covering all three paths (Flex / cron+agent / Event API),
   parameterized per signal.
4. Produce at least one cross-signal diagnosis (dependency-graph collapse) with dwell-based onset.

## Implementation rules

- **Provado stays remote.** No direct MySQL/SSH/host access from Provado; signals arrive only as
  `ProvadoSignal` events read via NerdGraph. (The lab shippers we set up run on the Magento host;
  Provado itself only reads New Relic.)
- **Ship raw state; diagnose in Provado.** Shippers emit current-state snapshots; dwell, baselines,
  correlation, and deploy-stamping live in Provado. Do not push verdicts from the shipper.
- **Reuse the v0.1.0–v0.4.0 seams** (`HttpClient`, `NerdGraphClient`, canonical `Signal`,
  `CorrelationEngine`, `DiagnosticPattern`, `SourceAdapter`); the `ProvadoSignal` reader is a path
  on the existing NerdGraph client, not a new transport.
- **No secrets in the repo.** Shipper templates use placeholders; live credentials stay in the
  server `.env` (gitignored).
- **Stay narrow first:** one signal end-to-end before breadth; one correlation pattern before a
  library.
- Every source-code phase is green on the lab server + a `/code-review` pass (per `CLAUDE.md`).

## Phase 1 — Generic `ProvadoSignal` reader

**Status:** [x]

**Goal:** Read `ProvadoSignal` custom events from New Relic and map them to canonical signals,
regardless of which shipper produced them.

> Core source/mapping change — **plan approval required** per `CLAUDE.md`.

**Deliverables:**

- A NerdGraph query path (`SELECT * FROM ProvadoSignal WHERE signal = ... SINCE ...`) on the
  existing `NerdGraphClient`, selectable by config (e.g. a `mode: operational_signals` option)
  alongside the existing transaction-health query.
- A generic custom-event → canonical `Signal` mapper: `signal` → type, `source` → source, configured
  entity-attribute names → entity references, numeric fields → attributes, event timestamp → signal
  timestamp.
- Unit + contract tests over a redacted `ProvadoSignal` recording (no live calls).

## Phase 2 — Shipper toolkit + `cron_health` Wire-Up (live)

**Status:** [x]

> Live-validated: the php-agent shipper emits `cron_health` to the lab's New Relic and
> `provado:signals` (operational mode) reads it back — `error=2855 missed=0 pending=378
> running=0 success=1455`, entity `host:provado`. (Lab runner note: testbench loads
> `vendor/orchestra/testbench-core/laravel/.env`, symlinked to the package `.env` so the
> `mode` is picked up; in a real host app the app `.env` is read normally.)

**Goal:** Ship the first real operational signal from the lab and read it end-to-end.

**Deliverables:**

- Versioned shipper templates for all three paths (Flex YAML, Magento cron + `newrelic_record_custom_event`,
  Event API script), parameterized per signal, under `docs/shippers/` (or a `shippers/` dir).
- The `cron_health` signal definition (source query + `ProvadoSignal` shape).
- A live Wire-Up on the lab (one shipper) emitting `cron_health` to New Relic, surfaced in
  `provado:signals` — the end-to-end validation.

## Phase 3 — `indexer_status` signal

**Status:** [ ]

**Goal:** Ship indexer/mview staleness (`mview_state` vs `MAX(*_cl.version_id)`, `indexer_state`),
including the valid-while-failed case (ACSD-51431), and read it as a canonical signal.

## Phase 4 — `queue_backlog` signal (+ RabbitMQ liveness)

**Status:** [ ]

**Goal:** Ship message-queue backlog/liveness (`queue_message_status` + RabbitMQ `/api/queues`
ready/unacked/consumers/ack-rate) and read it; distinguish alive-but-stalled from disconnected.

## Phase 5 — Config/deploy spine (`config_change` + NR deploy markers)

**Status:** [ ]

**Goal:** Build the change timeline: ship `core_config_data.updated_at` deltas as `config_change`,
and read New Relic deploy/change markers (`FROM Deployment`) on the NerdGraph client. This is the
spine every other diagnosis is stamped against.

## Phase 6 — First dependency-graph diagnosis + dwell baseline

**Status:** [ ]

**Goal:** The first real diagnosis on top of ingestion: a pattern that collapses correlated
operational signals by Magento's dependency graph (e.g. cron death → cache/index/email staleness)
into one verdict, alarming on **dwell** (how long the contradiction persists) rather than instant
state, and stamping the onset against the deploy/config timeline.
