## Version goal

**Turn the raw operational-signal stream into a learned, dwell-aware diagnosis.** Reduce a window
of snapshots to current state per `(signal, entity)`; alarm on *how long* a contradiction has
persisted (real onset-based dwell, not a `maxâˆ’min` proxy); learn what normal looks like per metric
instead of hard-coding thresholds; and collapse by Magento's explicit dependency graph, including
detecting a cron worker stuck in `running`.

The shipper stays dumb (raw state only); all of this new intelligence lives in Provado, on top of
the v0.5.0 ingestion seams (`SignalStore`, `CorrelationEngine`, `DiagnosticPattern`).

## v0.6.0 success definition

The system can, against the live lab:

1. Ship the operational signals **continuously** (scheduled, not on-demand), so the read window
   holds a real time series rather than a single hand-run snapshot.
2. Report **true onset-based dwell** for the cron signal (`cron_dwell_seconds` reflects how long
   cron has been degraded, computed by walking the series back to the onset) â€” not `0`, not
   `maxâˆ’min`.
3. Fire `CronHealthPattern` on **deviation from a learned-normal baseline** (per-metric, from the
   series) rather than from a hard-coded threshold, with the static bound kept only as a safety
   floor.
4. Detect a **stuck-`running`** cron worker as a fire condition, and collapse downstream symptoms
   by an **explicit dependency graph** (cron upstream of cache / index / email / queue), lighting
   up the symptoms for the signals we ship today and declaring the edges for the ones we do not yet.

## Implementation rules

- **Provado stays remote.** No new direct access; the only new collection is scheduling the
  existing merchant-side shipper. Everything else reads `ProvadoSignal` via NerdGraph as before.
- **Ship raw state; diagnose in Provado.** Dwell, baselines, and graph collapse are added on the
  Provado side only â€” the shipper is unchanged.
- **Reuse the v0.1.0â€“v0.5.0 seams** (`SignalStore`/`SignalQuery`, `CorrelationEngine`,
  `CorrelationGroup`, `DiagnosticPattern`, `CronHealthPattern`). Dwell/baseline are new helpers on
  these seams, not a new transport or a parallel engine.
- **Stay narrow, one thing at a time.** Reduction + real dwell before baselines; baselines before
  the fuller graph. Don't build a baseline library before one learned metric works end to end.
- **Honest scope on the dependency graph.** Encode the full graph structure, but only activate
  downstream symptoms for signals that actually ship today (`indexer_status`, `queue_backlog`).
  `cache_validity` and an email/queue-consumer signal are deferred (they need the "Instrument"
  shipper) â€” they remain declared graph edges that light up when those signals exist.
- Every source-code phase is green on the lab server + a `/code-review` pass (per `CLAUDE.md`).

## Phase 1 â€” Continuous shipping

**Status:** [ ]

**Goal:** Make the operational signals flow continuously so the read window holds a real time
series â€” the precondition for any dwell computation. Operational + docs only; no `src/` change.

**Deliverables (items):**

1. Schedule `shippers/php-agent/provado-ship.php` on the lab via OS-cron (every 5 minutes, NR PHP

   `ProvadoSignal` events keep arriving across several intervals via `provado:signals`.
2. Update the lab/scheduling notes in `docs/shipping-methods.md` and `shippers/README.md` from
   "run on demand during development" to the scheduled reality, recording the exact cron entry.

## Phase 2 â€” Dwell foundation

**Status:** [ ]

> Core diagnosis-layer change â€” **plan approval required** per `CLAUDE.md`.

**Goal:** Fix the two structural problems behind `cron_dwell_seconds = 0`: the window holds many
per-poll snapshots per `(signal, entity)`, and dwell is a within-group `maxâˆ’min` proxy. Reduce to
current state, then compute dwell as the time since the *onset* of the current state.

**Deliverables (items):**

1. A **latest-per-`(signal, entity)` reduction** helper over a set/window of signals, so patterns
   reason over current state instead of a pile of duplicate snapshots. Unit-tested over a
   multi-poll series.
2. **Onset-based dwell:** given a `(signal, entity)` series and a "is this state bad?" predicate,
   walk the series back to find when the current bad state began and return the elapsed seconds.
   Re-point `CronHealthPattern`'s `cron_dwell_seconds` at this so it reports true dwell. Live-verify
   against the now-continuous lab series (Phase 1).

## Phase 3 â€” Learned-normal baselines

**Status:** [ ]

> Core diagnosis-layer change â€” **plan approval required** per `CLAUDE.md`.

**Goal:** Replace the hard-coded `PENDING_BACKLOG = 100` / `ERROR_ELEVATED = 500` with a baseline
learned from the series, so slow drift is caught relative to normal â€” the architecture doc's
"learn normal â€¦ so a slow drift is caught before it trips a hard limit."

**Deliverables (items):**

1. A **baseline computation** per `(signal, entity, metric)` from the series (e.g. a robust
   central tendency + spread over a baseline window â€” median/percentile, not mean, to resist
   spikes). Unit-tested for the drift and the spike cases.
2. `CronHealthPattern` fires on **deviation from baseline** instead of the static threshold, with
   the static bound retained only as a floor so a cold start (no baseline yet) still alarms on an
   egregious value. Live-verify on the lab.

## Phase 4 â€” Full dependency graph + stuck-`running`

**Status:** [ ]

> Core pattern change â€” **plan approval required** per `CLAUDE.md`.

**Goal:** Make the dependency-graph collapse explicit and dwell-aware, and add the cron failure
shape v0.5.0 omitted: a worker stuck in `running`.

**Deliverables (items):**

1. **Stuck-`running` detection** as a fire condition: `running > 0` whose dwell exceeds the normal
   job runtime (from the Phase 3 baseline) is a dead worker, distinct from `missed` (scheduler not
   firing) and `pending` (backlog). Per the catalog, "an indexer/cron in `working`/`running` can be
   frozen, not busy."
2. An **explicit dependency graph** (cron upstream of cache / index / email / queue) replacing the
   ad-hoc inline checks in `downstreamSymptoms()`. Activate symptoms for the shipped signals
   (`indexer_status`, `queue_backlog`), dwell-qualified; declare the `cache`/`email` edges as
   awaiting their signals (deferred). Live-verify the collapsed verdict on the lab.
