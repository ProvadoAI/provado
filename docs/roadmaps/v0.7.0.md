## Why now / why this shape

- The two dark edges are **already built**. v0.6.0 Phase 4 encodes the full graph including the
  `cache` and `email` edges; it only declines to *activate* them for lack of a feeding signal.
  v0.7.0 supplies the signals — it does **not** re-architect the graph. When each signal arrives,
  the corresponding edge lights up against infrastructure that already exists (dwell, baselines,
  collapse).
- These two signals are **"Instrument"**, not **"Wire-Up"** (`ARCHITECTURE_DIRECTION_SOURCE.md`,
  `signal-shipping.md` §catalog). The v0.5.0 signals are raw `SELECT`s the shipper runs over the
  DB. `cache_validity` needs `Magento\Framework\App\Cache\TypeListInterface::getInvalidated()`, and
  email staleness needs **consumer liveness** (is the queue-consumer process actually alive, not
  just how deep the queue is). Neither is a single SQL query, so both require a shipper that
  **bootstraps the Magento application** to reach its internal APIs — the capability the current
  `php-agent` shipper (raw PDO over `app/etc/env.php`) does not have. That shared capability is the
  first phase; the two signals build on it.
- Completing the lead pattern is the point. The architecture doc's v1 lead pattern is *"cron death →
  cache/index/email staleness collapse"*. v0.6.0 proves the cron→index/queue half on the lab;
  v0.7.0 closes the cache and email halves so the full collapse is demonstrable end to end.

## Version goal

**Light up the two dark dependency-graph edges (`cron→cache`, `cron→email`) by shipping the two
"Instrument" signals that feed them, and complete the v1 lead pattern.** Build the Magento-bootstrap
shipper capability once, ship `cache_validity` and a consumer-liveness signal on top of it, and
activate the corresponding edges in Provado — reusing the v0.6.0 dwell/baseline/graph seams, with no
new transport and no graph redesign.

## v0.7.0 success definition

The system can, against the live lab:

1. Run an **"Instrument" shipper** that bootstraps Magento and emits two new `ProvadoSignal` types
   continuously (same schedule/contract as the v0.5.0–v0.6.0 signals), shipping **raw state only**.
2. Ship **`cache_validity`** (which cache types are invalidated, per `getInvalidated()`), and have
   Provado **activate the `cron→cache` edge** — a cache left invalidated while cron is degraded
   collapses under the cron root cause, **dwell-qualified** by the v0.6.0 dwell/baseline layer
   rather than firing as its own independent alarm.
3. Ship a **consumer-liveness** signal (a message-queue consumer that should be running but is
   dead/stalled — the silent-consumer-death failure mode), and have Provado **activate the
   `cron→email` edge** so stale transactional email collapses under the same root cause.
4. Demonstrate the **complete v1 lead pattern** on the lab: a single degraded-cron condition
   collapsing `index` + `queue` (from v0.6.0) **and** `cache` + `email` (new) into one root-cause
   verdict with downstream symptoms attached — not four separate alarms.

## Implementation rules

- **Provado stays remote.** No new direct access from Provado. The only new merchant-side
  collection is the "Instrument" shipper running on the Magento host, same as the existing shippers.
  Provado reads `ProvadoSignal` via NerdGraph exactly as before.
- **Ship raw state; diagnose in Provado.** The Instrument shipper emits raw invalidation lists and
  raw consumer status. Dwell, baselines, edge activation, and collapse stay on the Provado side,
  reusing the v0.6.0 seams.
- **No graph redesign.** The `cache` and `email` edges already exist in the Phase-4 graph. v0.7.0
  activates them by supplying signals; it must not rebuild `downstreamSymptoms()` or the graph
  structure. If an edge needs a tweak to light up, that is a small change to an existing edge, not a
  new engine.
- **Reuse v0.6.0 dwell/baseline.** New edge activations are **dwell-qualified** and baseline-aware
  using the helpers v0.6.0 ships — do not reintroduce static thresholds for the new signals beyond a
  safety floor.
- **Stay narrow, one thing at a time.** Build the Instrument capability before any new signal; ship
  and activate one signal fully (cache) before the next (email); verify the complete lead pattern
  last. Don't ship both signals before the first edge lights up end to end.
- **Honest lab caveat.** The lab is Magento Open Source on-prem. `cache_validity` is observable
  there (cache invalidation happens normally). Consumer-liveness is observable only if the lab
  actually runs the relevant queue consumers — if it does not, ship the signal but state plainly that
  the `cron→email` collapse is verified by a forced/synthetic dead-consumer condition, not organic
  traffic (mirrors v0.6.0's honesty note on deploy markers).
- Every source-code phase is green on the lab server + a `/code-review` pass (per `CLAUDE.md`).

## Phase 1 — "Instrument" shipper foundation

**Status:** [ ]

> Shipper/collection change (merchant-side, not Provado `src/` core). Plan the bootstrap approach
> before implementing; treat the existing `php-agent` shipper as the base to extend, not replace.

**Goal:** Give the shipper the one capability both new signals need and the v0.5.0 signals never
did — reaching Magento's **internal APIs** (not just raw DB tables). The current
`shippers/php-agent/provado-ship.php` opens a raw `PDO` from `app/etc/env.php`; `getInvalidated()`
and consumer status are not DB rows, so the shipper must **bootstrap the Magento application** (load
`app/bootstrap.php`, obtain the object manager) to call them. This phase adds that capability and
ships nothing new yet — it is the precondition for Phases 2–3.

**Deliverables (items):**

1. An **Instrument-capable shipper path** that bootstraps Magento and exposes the object manager to
   the signal functions, while keeping the existing raw-SQL signals working unchanged (cron,
   indexer, queue, config). Decide and document: extend `provado-ship.php` vs. a sibling
   `provado-ship-instrument.php` (a heavier bootstrap may warrant its own less-frequent schedule).
2. Update `docs/shipping-methods.md`, `docs/signal-shipping.md`, and `shippers/README.md` to record
   the **Wire-Up vs. Instrument** distinction concretely (what each new signal reads, why it needs
   bootstrap), and the scheduling/cost note for the Instrument path on the lab.

## Phase 2 — `cache_validity` signal end-to-end

**Status:** [ ]

> Core pattern/edge change on the Provado side — **plan approval required** per `CLAUDE.md`.

**Goal:** Ship the first Instrument signal and light up the first dark edge. `cache_validity`
reports which cache types are currently invalidated (`getInvalidated()`); Provado activates the
`cron→cache` edge so an invalidated cache **while cron is degraded** collapses under the cron root
cause instead of firing standalone.

**Deliverables (items):**

1. **Ship `cache_validity`** from the Instrument shipper: raw state per the catalog
   (`invalidated_count`, and the invalidated type list as entities) — no verdict. Per
   `signal-shipping.md` §catalog. Verify it arrives continuously on the lab.
2. **Activate the `cron→cache` edge** in Provado: the edge declared (dark) in v0.6.0 Phase 4 now
   reads `cache_validity`, **dwell-qualified** via the v0.6.0 dwell layer, and the cache symptom
   collapses under a degraded-cron root cause. Reuse the existing graph; do not rebuild it.
   Unit-tested for the collapse-vs-standalone cases; live-verify the collapsed verdict on the lab.

## Phase 3 — Consumer-liveness (email) signal end-to-end

**Status:** [ ]

> Core pattern/edge change on the Provado side — **plan approval required** per `CLAUDE.md`.

**Goal:** Ship the second Instrument signal — **consumer liveness**, the silent-consumer-death
failure mode the catalog flags as straddling Instrument — and light up the `cron→email` edge. This
is distinct from `queue_backlog` (which reads queue *depth*): liveness is whether the consumer
process that drains the queue (e.g. the transactional-email consumer) is actually alive. A dead
consumer with cron degraded is the email-staleness half of the lead pattern.

> **Boundary with the v0.6.0 `queue` edge — read before implementing.** The `cron→queue` edge
> lit up in v0.6.0 Phase 4 already infers a dead consumer *indirectly*, from queue depth:
> `ready > 0 && consumers == 0` (`CronHealthPattern::downstreamSymptoms()`). The `cron→email`
> consumer-liveness signal detects the dead consumer *directly* (the process is gone / not acking),
> which is more precise but **overlaps** that condition. Keep the two edges from double-counting the
> same dead consumer: the `queue` edge keys on **queue depth** (a backlog nobody is draining), the
> `email`/consumer-liveness edge keys on **process liveness** (a consumer that should be running and
> is not). When both fire for the same consumer, they must collapse into the one cron root cause as a
> single attributed symptom, not two. This boundary is the reason the email edge is its own signal
> rather than a tweak to the queue edge — make it explicit in the edge predicates.

**Deliverables (items):**

1. **Ship the consumer-liveness signal** from the Instrument shipper: raw state per consumer (alive
   vs. expected-but-absent, last-activity / ack progress) — no verdict. Verify on the lab (forced
   dead-consumer condition if the lab does not organically run the relevant consumer — per the
   honest lab caveat).
2. **Activate the `cron→email` edge** in Provado: a dead/stalled email consumer collapses under a
   degraded-cron root cause, dwell-qualified, reusing the v0.6.0 graph. Distinguish a
   cron-caused stall (collapse) from an independently dead consumer (standalone). Unit-tested;
   live-verify on the lab.

## Phase 4 — Lead-pattern completion

**Status:** [ ]

> Verification/integration phase — proves the v1 lead pattern end to end. No new collection.

**Goal:** Demonstrate the complete architecture-doc lead pattern on the live lab: one degraded-cron
condition collapsing **all four** downstream symptoms — `index` + `queue` (v0.6.0) and `cache` +
`email` (v0.7.0) — into a single root-cause verdict with the symptoms attached, rather than four
independent alarms.

**Deliverables (items):**

1. **End-to-end lead-pattern verification** against the lab: induce/observe a degraded-cron window
   and confirm Provado emits one cron root-cause verdict that collapses cache + index + email +
   queue, each dwell-qualified. Capture the verdict as evidence (as Phase 1 of v0.6.0 captured the
   shipping run). Document the result and tighten any edge whose collapse is shaky.
