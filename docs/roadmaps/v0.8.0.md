## Why now / why this shape

v0.7.0 marked the v1 lead pattern *"cron death → cache/index/email/queue staleness collapse"*
as complete and "verified end-to-end on the lab". The 2026-07-01 analysis raised a doubt about
that claim — framed as a **hypothesis to test**, not a settled fact:

- Correlation groups signals **only by shared entity**, with time-proximity **off by default**
  (`CorrelationEngine::correlate($window)` is called with no criteria).
- `cron_health` ships with **no entity at the shipper**; each downstream signal (`indexer_status`,
  `queue_backlog`, `cache_validity`, `consumer_liveness`) carries a **different** entity.
- **Hypothesis (feared):** so in production `cron_health` would be a singleton group → discarded
  (`groupsFor` needs `count > 1`), and the four-edge collapse — seen only under the tests' shared
  `host:provado` — would not reproduce on real shipper output.

**Phase 1 tested that hypothesis on the live lab and OVERTURNED it.** The collapse fires
**organically**: the **New Relic PHP agent auto-stamps `host`** on every custom event, and the
reader's `signal_entity_fields` already maps `host`→entity, so all signals from one host share
`host:provado` and correlate. The tests' `host:provado` was **never synthetic** — it mirrors what
the agent emits, so the v0.7.0 "verified" claim was correct for the PHP-agent shipper (see
"Phase 1 item 1" below).

**So this version is no longer a break-fix. It is (1) hardening — make the shared instance entity
INTENTIONAL rather than a New Relic-agent side-effect, and PORTABLE across all three shippers (the
Event-API shipper C sends no auto-`host`, so it would not collapse today); and (2) honest coverage
— reconcile the fixtures/catalog with real shipper output.**

## Version goal

**Phase 1 proved the collapse works organically (via the NR agent's `host`). This version now
guarantees it independent of that side-effect: give every `ProvadoSignal` an intentional shared
instance entity so all three shippers collapse by design — with no new transport and no graph
redesign — and reconcile the fixtures/catalog with real shipper output.**

## v0.8.0 success definition

Against the live lab, with the **real** shippers (not synthetic events):

1. A genuinely degraded cron condition produces **one** `CronHealthPattern` verdict collapsing
   the four downstream edges — reproduced with `provado:diagnose` over live NerdGraph, **without**
   any hand-injected shared entity. **✅ Done in Phase 1 (via the NR agent's auto-`host`).**
2. Every `ProvadoSignal` carries a common instance entity so signals from one Magento instance
   correlate by design. **✅ Done in Phase 2: both shippers stamp `source_instance` on every event;
   verified landing in NR for all six signal types (see "Phase 2 item 2 — re-verification result").**
3. The contract fixtures and the `signal-shipping.md` catalog match what the shippers actually
   emit (no invented `store`/`stuck`, no `ack_rate` claimed-but-unshipped).

## Implementation rules

- **Provado stays remote.** No new direct access; the only change on the merchant side is what
  the existing shippers put on each event.
- **No graph redesign.** The dependency graph and collapse logic are unchanged; this version fixes
  *what lands in the correlation group*, not how the group is diagnosed.
- **Reuse the entity mechanism.** `signal_entity_fields` already lists `host` — lighting it up is
  a shipper change plus config, not a new reader.
- Every source-code phase is green on the lab server + a `/code-review` pass (per `CLAUDE.md`).

### Phase 1 item 1 — verification result (2026-07-01, lab `provado`)

**Answer: the lead pattern collapses ORGANICALLY. The E1 premise is empirically wrong for the
PHP-agent shipper.** Run against the live lab with the real scheduled shippers (Wire-Up every 5 min,
Instrument every 15 min — both already in the host crontab), a genuinely degraded cron
(`missed=60`, `pending≈365–523`, `error=1448` — `missed` injected per `SIM-CRON-001`, error/pending
pre-existing), and `operational_signals` enabled, `provado:diagnose` over live NerdGraph produced
**one** `critical` `CronHealthPattern` verdict folding the downstream edges into it — **no
hand-injected shared entity**:

> Cron is degraded (523 pending, 60 missed, 1448 errored cron rows); the following downstream
> symptoms are attributed to it: cache config, cache layout, cache block_html, indexer
> catalog_product_price, indexer catalogsearch_fulltext.

All six signal types (`cron_health` + `indexer_status` + `queue_backlog` + `cache_validity` +
`consumer_liveness` + `config_change`) landed in **one** correlation group, joined by a shared
`host:provado` entity; 5 downstream symptoms (2 indexer + 3 cache edges) collapsed into the single
finding.

**Why it collapses — and why this is not what the roadmap assumed:**

- The shipper's `ship()` payload carries **no** entity on `cron_health` — verified from the raw
  `shipped:` line: `{"signal":"cron_health","source":"magento","pending":…,"missed":60,"error":1448}`.
  So the roadmap's "cron_health ships with no entity" is literally true **at the shipper**.
- But the **New Relic PHP agent auto-stamps `host` (plus `appId`/`appName`/`entityGuid`/`realAgentId`/
  `tags.*`) on every `newrelic_record_custom_event`.** The raw NR event for `cron_health` carries
  `host:provado`; `SELECT uniques(host) FROM ProvadoSignal FACET signal` returns `provado` for **all
  six** signal types.
- Provado's `signal_entity_fields` already lists `host`, so the reader maps that agent-injected
  `host` into a `host` entity. Every signal from the one host therefore shares `host:provado` and
  correlates — `cron_health` stops being a singleton and enters the group. **The test harness's
  `host:provado` was not synthetic; it mirrors what the agent actually emits in production.** The
  v0.7 "verified end-to-end" claim was correct for this (the PHP-agent) shipper.

**Remaining risk (why Phase 2 still has value — its rationale changes):** the shared entity is a
**side-effect of Shipper B (the NR PHP agent)**, not of the `ProvadoSignal` contract. Shipper C
(Event API POST) ships only the explicit payload — no auto-`host` — so the collapse would **not**
happen there; Shipper A (Flex) depends on the infra agent. So the organic collapse today is
**shipper-dependent and incidental**, not contractually guaranteed. Phase 2 (explicitly ship a
stable instance entity from **all** shippers) converts an incidental agent side-effect into an
intentional, shipper-independent guarantee — hardening something that already works for the default
Adobe Commerce Cloud shipper, not repairing a total break. **Recommend re-reading Phase 2's framing
with this in mind before starting it.**

### Phase 1 item 3 — config posture decision (2026-07-01)

**Decision: keep `operational_signals` OPT-IN for v0.8.0, document it explicitly, and flip the
default to include it at the end of Phase 2.** No behavior change now — `PROVADO_NEW_RELIC_MODES`
still defaults to `transaction_health`; the change is explicit docs (`config/provado.php` comment,
`docs/signal-shipping.md`).

Rationale:

- The flagship diagnosis lives in `operational_signals`, and P1 item 1 proved it collapses
  organically — but today that relies on an **incidental** NR-agent side-effect (auto-`host`) and
  only for Shipper B. Defaulting the flagship ON before Phase 2 makes the instance entity
  intentional would promote behavior that is not yet contractually guaranteed across shippers.
- `operational_signals` reads the `ProvadoSignal` custom event type, which exists **only if the
  merchant deployed a shipper**. Defaulting it ON for a merchant without shippers reads an empty
  type and yields no findings — indistinguishable from "healthy," which is misleading. Opt-in keeps
  enabling it a deliberate act paired with deploying the shippers.
- The current default (`transaction_health`) feeds no pattern today (E4 — the symptom layer is
  produced live but unconsumed), so neither mode auto-produces a symptom diagnosis; that is E4's
  problem (v0.9.0), not a reason to flip the operational default prematurely.
- **Follow-up (Phase 2 close):** change the default to `transaction_health,operational_signals` as a
  one-line config change behind its own tests + `/code-review`, so the flagship is on by default
  exactly when Phase 2 guarantees the shipper-independent instance entity.

## Phase 2 — C1: shared instance entity

**Status:** [x]

> Core ingestion/correlation change — **plan approval required** per `CLAUDE.md`.

**Goal:** Give every `ProvadoSignal` a common instance entity (e.g. `host` / `source_instance`)
so all operational signals from one Magento instance share it, and `cron_health` collapses with its
downstream edges organically — not just under the test-injected `host:provado`.

**Deliverables (items):**

1. [x] **Ship the instance entity** from both shippers (`provado-ship.php` and
   `provado-ship-instrument.php`): add a `host`/`source_instance` attribute to every event, derived
   from a stable per-instance id. Reuse the existing entity-field mapping in the reader
   (`signal_entity_fields` already lists `host`); no new reader path.
   **→ Done: both shippers stamp `source_instance` on every event (resolved once in `ship()` from
   `PROVADO_INSTANCE` → hostname → source); `source_instance` added to
   `NerdGraphClient::DEFAULT_SIGNAL_ENTITY_FIELDS`. Chose `source_instance` over `host` so it is
   explicit and shipper-independent (no collision with the NR agent's intrinsic `host`, works for the
   Event API shipper). Docs updated in `signal-shipping.md`.**
2. [x] **Re-verify** the Phase-1 degraded-cron scenario now collapses organically, and update unit +
   contract tests so the fixtures reflect the real shipped shape (not a separately-injected entity).
   **→ Done. See "Phase 2 item 2 — re-verification result" below. Tests: mapper test
   (`source_instance`→entity, not a metric); correlation test (signals sharing ONLY `source_instance`,
   with disjoint own entities and no `host`, group into one — the Shipper C guarantee); contract
   recording + `LiveClientContractTest` now carry `source_instance` on both events and assert the
   shared instance entity. Suite green on the lab (322 tests, 903 assertions).**

### Phase 2 item 2 — re-verification result (2026-07-01, lab `provado`)

**Answer: the instance entity is now shipped intentionally and lands in New Relic for every signal,
independent of the agent's `host`.** Both real shippers were run against the live Magento install on
the lab (degraded cron still injected — `missed=60`, `error≈1406`, `pending≈340`):

- **Every event carries `source_instance`.** The Wire-Up shipper stamps it on `cron_health`,
  `indexer_status`, `queue_backlog`, `config_change`; the Instrument shipper on `cache_validity`,
  `consumer_liveness` — verified from the raw `shipped:` lines
  (`{"signal":"cron_health",…,"source_instance":"provado",…}`).
- **It reaches New Relic and is shared across all six signal types.**
  `SELECT uniques(source_instance) FROM ProvadoSignal FACET signal SINCE 15 minutes ago` over live
  NerdGraph returns `["provado"]` for **all six** facets (`cron_health`, `indexer_status`,
  `queue_backlog`, `cache_validity`, `consumer_liveness`, `config_change`). The reader maps
  `source_instance` (now in `DEFAULT_SIGNAL_ENTITY_FIELDS`), so all six correlate by design.

**Why this is the Phase-2 guarantee, not just a repeat of Phase 1:** in Phase 1 the collapse rode on
the NR agent's auto-`host` (Shipper B only). Now the shared entity is an explicit `ProvadoSignal`
attribute every shipper emits — so the Event-API shipper (C), which gets no auto-`host`, collapses
too. On the PHP-agent lab a signal now shares *both* `host:provado` and `source_instance:provado`
(harmless overlap); the collapse into one `CronHealthPattern` verdict is the same mechanism proven
organically in Phase 1 item 1, now anchored on the intentional entity. The Shipper-C path (grouping
on `source_instance` alone, with `host` absent and own entities disjoint) is covered by the new
`CorrelationTest` case.
