## Why now / why this shape

v0.7.0 marked the v1 lead pattern *"cron death → cache/index/email/queue staleness collapse"*
as complete and "verified end-to-end on the lab". The 2026-07-01 analysis found that verification
rests on data that the **real shippers do not emit**:

- Correlation groups signals **only by shared entity**, with time-proximity **off by default**
  (`CorrelationEngine::correlate($window)` is called with no criteria).
- `cron_health` ships with **no entity**; each downstream signal (`indexer_status`,
  `queue_backlog`, `cache_validity`, `consumer_liveness`) carries a **different** entity.
- So in production `cron_health` is a **singleton group → discarded** (`groupsFor` needs
  `count > 1`), and the four-edge collapse has nothing to collapse onto.
- Unit tests collapse because every signal also carries a shared `host:provado`
  (`CronHealthPatternTest`), and the Phase-4 synthetic scenario almost certainly injected one
  too. The recorded lab response `provado_signal_operational.json` even invents a `store` on
  `cron_health` that the real shipper never sends — evidence the "does it correlate?" question
  was never nailed down against real output.

**This version makes the lead pattern collapse on organic production signals, and gets the
coverage story honest before any breadth is added on top.**

## Version goal

**Prove (or disprove) the lead-pattern collapse on real shipper output, then guarantee it by
giving every operational signal a shared instance entity — with no new transport and no graph
redesign.** Reconcile the fixtures/catalog with reality so tests stop asserting a shape the
shippers don't produce.

## v0.8.0 success definition

Against the live lab, with the **real** shippers (not synthetic events):

1. A genuinely degraded cron condition produces **one** `CronHealthPattern` verdict collapsing
   the four downstream edges — reproduced with `provado:diagnose` over live NerdGraph, **without**
   any hand-injected shared entity.
2. Every `ProvadoSignal` carries a common instance entity so signals from one Magento instance
   correlate by design.
3. The contract fixtures and the `signal-shipping.md` catalog match what the shippers actually
   emit (no invented `store`/`stuck`, no `ack_rate` claimed-but-unshipped).

## Implementation rules

- **Provado stays remote.** No new direct access; the only change on the merchant side is what
  the existing shippers put on each event.
- **No graph redesign.** The dependency graph and collapse logic are unchanged; this version fixes
  *what lands in the correlation group*, not how the group is diagnosed.
- **Reuse the entity mechanism.** `signal_entity_fields` already lists `host` — lighting it up is
  a shipper change plus config, not a new reader.
- Every source-code phase is green on the lab server + a `/code-review` pass (per `CLAUDE.md`).

## Phase 1 — Verify & honest baseline

**Status:** [ ]

> Verification + docs. Minimal `src/` change. This is action #0: it decides whether C1 (Phase 2)
> is fixing a real break or hardening something that already works.

**Deliverables (items):**

1. [ ] **Organic lead-pattern verification on the lab.** Schedule the real Wire-Up + Instrument
   shippers, force a genuinely degraded cron (per `SIM-CRON-001`'s injection design), enable
   `operational_signals` mode, and run `provado:diagnose`. Record plainly whether the four edges
   collapse into one verdict **without** any synthetic shared entity. Capture the raw result either
   way — this is the empirical answer to the v0.7 "verified" claim.
2. [ ] **Re-audit the coverage claims** for failure modes #1, #4, #5, #8 against the code (the #1
   over-optimism correction showed the original matrix was generous). Produce honest per-mode notes
   (what is truly diagnosed vs. only shipped vs. missing) to feed the pending coverage map.
3. [ ] **Decide the config posture** for `operational_signals`: today `modes` defaults to
   `transaction_health`, so the lead pattern is silent unless the merchant opts in. Decide whether
   to default it on or keep it opt-in with explicit docs, and record the decision.

## Phase 2 — C1: shared instance entity

**Status:** [ ]

> Core ingestion/correlation change — **plan approval required** per `CLAUDE.md`.

**Goal:** Give every `ProvadoSignal` a common instance entity (e.g. `host` / `source_instance`)
so all operational signals from one Magento instance share it, and `cron_health` collapses with its
downstream edges organically — not just under the test-injected `host:provado`.

**Deliverables (items):**

1. [ ] **Ship the instance entity** from both shippers (`provado-ship.php` and
   `provado-ship-instrument.php`): add a `host`/`source_instance` attribute to every event, derived
   from a stable per-instance id. Reuse the existing entity-field mapping in the reader
   (`signal_entity_fields` already lists `host`); no new reader path.
2. [ ] **Re-verify** the Phase-1 degraded-cron scenario now collapses organically, and update unit +
   contract tests so the fixtures reflect the real shipped shape (not a separately-injected entity).
