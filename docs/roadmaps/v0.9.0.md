## Why now / why this shape

v0.8.0 makes the **operational** family collapse (cron ↔ its downstream) by giving them a shared
instance entity. But two deeper things remain:

- Correlation is still **binary** — a signal is in a group or not; there is no notion of *how
  strongly* it belongs. Time-proximity exists only as an all-or-nothing join that is off by default
  because turning it on merges everything ("everything co-occurs with everything").
- The **symptom** family (`transaction_health`, `checkout_failure_rate`) and the **operational**
  family still never meet: a symptom of APM latency is never collapsed with its operational cause,
  which is the tesis-defining capability the product sells.

This version lifts correlation from "in the group / not" to "in the group, with confidence X", and
uses that to attribute a symptom to its cause.

## Version goal

**Turn correlation into weighted attribution, and make a symptom collapse with its operational
cause.** Keep entity-based grouping as the structural skeleton; add temporal proximity (with dwell
and change-stamp) as a confidence weight applied *after* grouping; extend the shared instance entity
to symptom signals so symptom and cause land in one group; and expose the correlation rules as
configuration with read-only run snapshots.

## v0.9.0 success definition

1. A downstream symptom whose onset is close to the cause onset is attributed with **high
   confidence**; one that was already bad long before is weighted **low** (likely coincidence) — not
   a flat "everything in the group counts".
2. A `transaction_health`/checkout symptom and its operational cause from the **same instance**
   land in one correlation group and produce **one** verdict with backend attribution.
3. The correlation criteria (entity dimensions, proximity threshold/weight, filters, window), the
   dependency-graph edges, and the dwell/baseline thresholds are **configurable**, and a run's
   groups + report + evidence can be **persisted read-only** for history/audit.

## Implementation rules

- **Grouping stays binary and entity-based** (the skeleton). Proximity is a **weight**, never a
  join criterion — it must not be able to merge unrelated signals.
- **Reuse the v0.6.0 dwell/baseline and the v0.8.0 shared-entity** work; do not build a parallel
  engine.
- **Do not hand-edit derived groups.** The config surface manages *rules* (inputs) and persists
  *snapshots* (read-only outputs) — never editable group memberships.
- Every source-code phase is green on the lab server + a `/code-review` pass (per `CLAUDE.md`).

## Phase 1 — Proximity-as-weight

**Status:** [ ]

> Core correlation/diagnosis change — **plan approval required** per `CLAUDE.md`. See memory
> `proximity-as-weight-not-join`.

**Goal:** Add temporal proximity as a **confidence weight** on attribution, applied *after* the
entity-based grouping, so the collapse ranks symptoms by how well their onset aligns with the cause
onset instead of treating every group member equally.

**Deliverables (items):**

1. [ ] A **proximity/confidence score** per attributed downstream symptom, combining temporal
   proximity to the cause onset with the existing per-symptom dwell and the change-stamp. Unit-tested
   for the near-onset (high), long-before (low), and no-baseline (cold-start) cases.
2. [ ] `CronHealthPattern` (and the collapse) **surface the score** on each attached symptom and use
   it to demote coincidental symptoms, without changing which signals are grouped.

## Phase 2 — Cross-family correlation

**Status:** [ ]

> Core correlation change — **plan approval required** per `CLAUDE.md`.

**Goal:** Make a symptom (`transaction_health` / `checkout_failure_rate`) collapse with its
operational cause. This is *"resolve the symptom New Relic already shows with the attribution from
the backend flag it never read"* — the core product promise.

**Deliverables (items):**

1. [ ] **Extend the shared instance entity (v0.8.0 C1) to symptom signals** so a symptom and an
   operational signal from the same Magento instance land in one correlation group.
2. [ ] **Attribute the symptom to the operational cause** using the Phase-1 weight (e.g. a checkout
   latency symptom whose onset aligns with a cron/cache degradation is attributed to it), emitting
   one verdict that carries both the symptom NR shows and the backend cause. Guard against the
   over-broad grouping the direction doc warns about (narrow the symptom set that enters the join).

## Phase 3 — Correlation-criteria config surface + run snapshots

**Status:** [ ]

> Config/persistence change. See memory `correlation-criteria-config-surface`.

**Deliverables (items):**

1. [ ] Expose `CorrelationCriteria` (entity dimensions to join on, proximity threshold/weight,
   source/type/severity filters, window), the `DependencyGraph` edges, and the dwell/baseline
   thresholds as **configuration** (today they are code-only).
2. [ ] Persist a **read-only snapshot** of a run (correlation groups + incident report + evidence)
   so a past diagnosis can be inspected later. Reuse the `DatabaseSignalStore` seam; do not make
   groups editable.
